# RabbitMQ Queue Policies

## Rationale

Plan 0066 deliberately deferred all RabbitMQ policy-style queue knobs (delivery limit, delayed retry, dead-lettering, TTL, length limits) to "operator policies out-of-band," and the README's "Resilience and redelivery" section still tells users to tune those knobs outside this API. After shipping the resilience slice, the operator-managed-topology / `rabbitmqctl` / management-API route was reconsidered: it would require a dedicated `rabbitmqctl` binary or management-plane credentials at the application, neither of which we want to depend on at runtime.

Instead, this plan brings those knobs into the existing `RabbitMqTopologyBuilder` / `RabbitMqQueueBuilder` API as queue declaration arguments (`x-` args), so the application's topology definition is the single Configuration-as-Code source for both resource existence and resource policy. This works in both declare scenarios:

- **Active declare** — the args are sent to the broker with `QueueDeclareAsync`, so the policy lives on the queue itself.
- **Passive declare** — the framework only asserts existence via `QueueDeclarePassiveAsync` (no args, no `406` risk); the configured args are still recorded on the definition, where they drive the compiler's quorum-only guards.

RabbitMQ actively redeclared queues throw `406 PRECONDITION_FAILED` when their arguments differ from what is already on the broker, which would make reconfiguration a startup exception. This plan does **not** introduce automatic physical-name migration (no evolutionary auto-naming, no broker-side manifest). Instead, it follows an explicit, developer-driven workflow:

1. **Introduce** a new queue (for example `orders-v2`) under a new name with the new args; keep `orders-v1` in code so it is re-declared unchanged (idempotent) or skipped. Add the new binding `ex → orders-v2` (Active) and flip the old binding `ex → orders-v1` to `Delete` mode so the broker unbinds it and new messages stop flowing to `orders-v1`. Move consumers to `orders-v2`. Deploy.
2. **Drain** `orders-v1` on the broker until empty (its binding is gone, so no new messages arrive).
3. **Remove** `orders-v1`: either the operator deletes it via the management UI, or the dev flips its declare mode to `Delete` and deploys once more (the framework calls `QueueDeleteAsync(ifUnused, ifEmpty)`, which fails safely with a clear "drain first" error if the queue is not yet empty). Then remove both the old queue and old binding lines entirely.

Because the dev picks a new name when args change, the framework never re-declares an existing queue with different args, so the `406` never surfaces at startup. The framework's responsibilities are therefore: (a) expose the knobs, and (b) provide the `Delete` declare mode for queues **and** the `Delete` binding mode for queue bindings so both creation and removal are Configuration-as-Code.

Exchanges are out of scope for this plan; the `Delete` declare mode is defined on the shared `RabbitMqDeclareMode` enum and the `Delete` binding mode on the shared `RabbitMqBindingMode` enum (so they read correctly for exchanges too), but no exchange builder surface, exchange-delete provisioner path, or exchange-binding-delete provisioner path is added here — an exchange in `Delete` mode or an exchange binding in `Delete` mode throws at provisioning time.

## Acceptance Criteria

- [ ] `RabbitMqQueueBuilder` gains typed, validating methods for the policy-style queue declaration arguments not already present: `WithDeliveryLimit(int)`, `WithDelayedRetry(RabbitMqDelayedRetryType, TimeSpan minDelay, TimeSpan? maxDelay = null)` (and an overload taking only the min delay), `WithDeadLetterStrategy(RabbitMqDeadLetterStrategy)`, `WithOverflow(RabbitMqOverflow)`, `WithMaxPriority(byte)`, `WithQueueLeaderLocator(RabbitMqQueueLeaderLocator)`, `WithInitialClusterSize(int)`, `WithConsumerTimeout(TimeSpan)`. Each writes the corresponding `x-` arg into the existing arguments dictionary and validates its input (non-negative counts, non-negative delay, defined enum, etc.), matching the style of the existing `WithDeadLetterExchange` / `WithMessageTtl` / `WithMaxLength` methods.
- [ ] A `RabbitMqDelayedRetryType` enum (`Disabled`, `All`, `Failed`, `Returned`), a `RabbitMqDeadLetterStrategy` enum (`AtMostOnce`, `AtLeastOnce`), a `RabbitMqOverflow` enum (`DropHead`, `RejectPublish`, `RejectPublishDlx`), and a `RabbitMqQueueLeaderLocator` enum (`ClientLocal`, `Balanced`) are added with XML doc comments, mirroring the existing `RabbitMqQueueType` / `RabbitMqPublisherConfirmMode` enums.
- [ ] In **Active** declare mode the new args are declared to the broker (the provisioner already passes `queue.Arguments` through); no provisioner change is required for the new knobs. In **Passive** and **Skip** modes the args are recorded on the definition but not sent to the broker (the provisioner already calls `QueueDeclarePassiveAsync` with no args, or does nothing).
- [ ] The compiler guards queue-type-incompatible knobs, hard-erroring with the queue name and a remediation, mirroring the existing `WithRedelivery` guard: (a) `WithDeliveryLimit`, `WithDelayedRetry`, `WithDeadLetterStrategy`, `WithQueueLeaderLocator`, `WithInitialClusterSize`, and `WithConsumerTimeout` on a `Classic` or `Unknown` effective queue type — these are quorum-only — pointing at `AsQuorumQueue()` / `QueueType(Quorum)`; (b) `WithOverflow(RejectPublishDlx)` on a `Quorum` effective queue type — quorum queues do not support `reject-publish-dlx` — pointing at `DropHead`/`RejectPublish` or `AsClassicQueue()`; (c) `WithMaxPriority` on a `Quorum` effective queue type — quorum queues always enable the full 0-31 priority range and silently **ignore** `x-max-priority`, so the user's configured range would not take effect — pointing at `AsClassicQueue()` to control the range. The effective queue type is resolved with the existing `ResolveConsumerQueueType` / `GetDeclaredQueueType` machinery (active `x-queue-type` else explicit `QueueType(...)` else `Unknown`); the guards also run in `ValidateQueueDefinitions` for queues that no consumer references.
- [ ] `RabbitMqDeclareMode` gains a `Delete` value. `RabbitMqTopologyProvisioner.ProvisionQueueAsync` calls `channel.QueueDeleteAsync(queue.Name, ifUnused: true, ifEmpty: true, cancellationToken)` for `Delete` mode and throws the broker's error unchanged when the queue is not empty or still has consumers (the "drain first" signal). A `404 NOT_FOUND` from a delete on an already-absent queue is treated as success (the desired state — resource absent — is already achieved), so `Delete` mode is idempotent across restarts. `ProvisionExchangeAsync` throws `ArgumentOutOfRangeException` for `Delete` (exchanges are out of scope).
- [ ] `RabbitMqBindingMode` gains a `Delete` value. `RabbitMqTopologyProvisioner.ProvisionBindingAsync` calls `channel.QueueUnbindAsync(binding.SourceExchangeName, binding.QueueName, binding.RoutingKey, arguments, cancellationToken)` for a `RabbitMqQueueBindingDefinition` in `Delete` mode; a `RabbitMqExchangeBindingDefinition` in `Delete` mode throws `ArgumentOutOfRangeException` (exchanges are out of scope). The unbind uses the binding's recorded `Arguments` so a headers-exchange binding is matched correctly. A broker not-found error on an already-absent binding is treated as success (idempotent across restarts).
- [ ] The provisioner's binding loop skips a binding whose destination queue is in `Delete` mode (RabbitMQ removes bindings automatically when a queue is deleted, so an explicit unbind is redundant and a re-bind would target a soon-to-be-absent queue); this skip applies regardless of the binding's own mode. A `Delete`-mode binding whose queue is `Active`/`Passive`/`Skip` is **not** skipped — that is the Update-1 unbind that stops new messages flowing to the old queue. The binding loop runs in **two passes to guarantee routing continuity**: all `Active` (and no-op `Skip`) bindings first, then all `Delete` bindings, regardless of the order the user writes them in the topology — so in Update 1 the new `ex → orders-v2` bind is established before the old `ex → orders-v1` unbind runs, and there is never a window in which the exchange has neither binding. Provisioning order stays exchanges → queues → bindings.
- [ ] The compiler validates `Delete` mode: a consumer referencing a `Delete`-mode queue is a hard error ("Inbound consumer for queue '…' references a queue declared with Delete mode"), and `Delete` is rejected where `Active`/`Passive`/`Skip` are currently accepted only by extending the existing `Enum.IsDefined` checks (no extra branches). `Delete` mode on an exchange definition, an exchange binding in `Delete` mode, and a `Delete`-mode queue binding whose destination queue is also `Delete`-mode are accepted by the enum/provisioner-skip but the first two surface at provisioning (no silent success).
- [ ] The README's "Resilience and redelivery" section is updated: the paragraph deferring delivery-limit / delayed-retry / dead-letter tuning to "RabbitMQ policies or operator-managed topology outside this API" is replaced with the new in-API knob methods and a short example; a new "Evolving topology resources" subsection documents the introduce → drain → `Delete` workflow (including that `Delete` fails safely when the queue is not empty) and notes that the management UI remains a valid alternative for removal.
- [ ] All newly public types and members carry XML doc comments (CS1591 is the Release gate; `TreatWarningsAsErrors` stays clean).
- [ ] Tests: unit tests for each new knob method (arg name and value written, invalid input rejected), the new enums, the queue-type-incompatible guards (quorum-only knobs on classic/unknown, `RejectPublishDlx` and `WithMaxPriority` on quorum, including the passive-but-`QueueType(Quorum)` path), `Delete`-mode compiler guards (consumer on a Delete queue, exchange Delete surfacing at provisioning, exchange binding Delete surfacing at provisioning). Sociable integration tests on the `RabbitMqFixture` (RabbitMQ 4.3.1) assert: (a) an Active queue with the new quorum knobs is declared and a `WithDeliveryLimit`-bounded retryable failure dead-letters at the configured limit; (b) introducing `orders-v2` alongside `orders-v1` with the old binding flipped to `Delete` declares both queues without `406`, unbinds `ex → orders-v1`, binds `ex → orders-v2`, and the consumer consumes from `orders-v2` while new publishes no longer land in `orders-v1`; (c) flipping `orders-v1` to `Delete` after it is empty removes it, while flipping to `Delete` while it still holds messages throws a clear error and leaves the queue intact; (d) re-provisioning a topology whose `Delete`-mode queue/binding are already absent on the broker succeeds (idempotent restart).
- [ ] Code Coverage should stay above 93%. Use Microsoft.Testing.Extensions.CodeCoverage to gather coverage values.

## Technical Details

### Queue builder knobs (`RabbitMqQueueBuilder`)

Each new method follows the existing `WithDeadLetterExchange` / `WithMessageTtl` pattern: validate input, write the `x-` arg into `_arguments`, return `this`. Arg names and value encodings (verified against the RabbitMQ 4.3 docs):

- `WithDeliveryLimit(int limit)` → `x-delivery-limit` (int). Reject `< -1`; `-1` disables the limit (not recommended).
- `WithDelayedRetry(RabbitMqDelayedRetryType type, TimeSpan minDelay, TimeSpan? maxDelay = null)` → `x-delayed-retry-type` (`disabled`/`all`/`failed`/`returned` string), `x-delayed-retry-min` (milliseconds, long), and `x-delayed-retry-max` (milliseconds, long) when `maxDelay` is supplied. An overload `WithDelayedRetry(TimeSpan minDelay, TimeSpan? maxDelay = null)` defaults type to `All`. Reject negative delays. Quorum 4.3+.
- `WithDeadLetterStrategy(RabbitMqDeadLetterStrategy strategy)` → `x-dead-letter-strategy` (`at-most-once`/`at-least-once`). Quorum 4.3+; `at-least-once` additionally requires `overflow = reject-publish` (a runtime/broker constraint, not enforced by the compiler — documented on the method).
- `WithOverflow(RabbitMqOverflow overflow)` → `x-overflow` (`drop-head`/`reject-publish`/`reject-publish-dlx`). `reject-publish-dlx` is classic-only (compiler guard below).
- `WithMaxPriority(byte maxPriority)` → `x-max-priority`. Reject `0`. Classic-only (compiler guard below — quorum queues silently ignore this arg and always use 0-31).
- `WithQueueLeaderLocator(RabbitMqQueueLeaderLocator locator)` → `x-queue-leader-locator` (`client-local`/`balanced`). Quorum-only (guarded).
- `WithInitialClusterSize(int size)` → `x-quorum-initial-group-size` (int). Reject `< 1`. Quorum-only (guarded).
- `WithConsumerTimeout(TimeSpan timeout)` → `x-consumer-timeout` (milliseconds, long). Reject non-positive. Quorum 4.3+ (guarded).

The existing generic `WithArgument(string, object?)` remains for anything not covered; the typed methods are convenience + validation.

### Enums

`RabbitMqDelayedRetryType` (`Disabled = 0`, `All`, `Failed`, `Returned`), `RabbitMqDeadLetterStrategy` (`AtMostOnce = 0`, `AtLeastOnce`), `RabbitMqOverflow` (`DropHead = 0`, `RejectPublish`, `RejectPublishDlx`), `RabbitMqQueueLeaderLocator` (`ClientLocal = 0`, `Balanced`). Each carries XML docs and a string mapping used by the builder methods (the mapping lives in the builder methods, not the enum, mirroring `RabbitMqQueueType` usage in `GetDeclaredQueueType`).

### Compiler guards

Add a `ValidateQueueKnobs(queue, effectiveQueueType, validationErrors)` helper called from `ValidateQueueDefinitions` for each queue (resolving the effective type the same way `ResolveConsumerQueueType` does for consumers, so queues without consumers are still guarded). Hard-error in three cases, each naming the queue, the offending arg, and the remediation, mirroring the existing classic-only-flags error:

1. A quorum-only arg (`x-delivery-limit`, `x-delayed-retry-*`, `x-dead-letter-strategy`, `x-queue-leader-locator`, `x-quorum-initial-group-size`, `x-consumer-timeout`) is present on Classic/Unknown → point at `AsQuorumQueue()` / `QueueType(Quorum)`.
2. `x-overflow = reject-publish-dlx` is present on Quorum → point at `DropHead`/`RejectPublish` or `AsClassicQueue()` (quorum queues do not support `reject-publish-dlx`).
3. `x-max-priority` is present on Quorum → point at `AsClassicQueue()` (quorum queues silently ignore this arg and always use 0-31 priorities, so the configured range would not take effect).

The existing `GetDeclaredQueueType` / `TryGetQueueTypeArgument` helpers are reused unchanged.

### `RabbitMqDeclareMode.Delete`

Add `Delete = 3` to the enum with XML docs describing the safe delete (ifUnused, ifEmpty) and the introduce → drain → delete workflow. The `RabbitMqDeclareMode` XML remarks update to mention `Delete`.

`RabbitMqTopologyProvisioner.ProvisionQueueAsync` gains a `Delete` arm calling `channel.QueueDeleteAsync(queue.Name, ifUnused: true, ifEmpty: true, cancellationToken: cancellationToken)`. A `404 NOT_FOUND` (queue already absent) is caught and treated as success — `Delete` mode is idempotent across restarts, so a topology that still lists a `Delete` queue after the broker has already removed it does not fail provisioning. Other broker errors (in particular the "drain first" failure when the queue is not empty or still has consumers) propagate unchanged. `ProvisionExchangeAsync` gains a `Delete` arm that throws `ArgumentOutOfRangeException` (exchanges out of scope).

### `RabbitMqBindingMode.Delete`

Add `Delete = 2` to the enum with XML docs describing the Update-1 unbind use case. `RabbitMqQueueBindingBuilder` / `RabbitMqExchangeBindingBuilder` already expose `WithBindingMode(RabbitMqBindingMode)`, so no builder method changes are needed — the new enum value flows through unchanged.

`RabbitMqTopologyProvisioner.ProvisionBindingAsync` gains `Delete` arms: for a `RabbitMqQueueBindingDefinition` call `channel.QueueUnbindAsync(binding.SourceExchangeName, binding.QueueName, binding.RoutingKey, arguments, cancellationToken)` using the binding's recorded `Arguments` (so a headers-exchange binding is matched); a broker not-found error on an already-absent binding is caught and treated as success (idempotent across restarts). For a `RabbitMqExchangeBindingDefinition` throw `ArgumentOutOfRangeException` (exchanges out of scope).

The binding loop in `ProvisionAsync` builds a `HashSet<string>` of `Delete`-mode queue names once, then iterates bindings in **two passes**: first all bindings whose own mode is `Active` or `Skip` (creating/re-asserting bindings, including the new `ex → orders-v2` bind), then all bindings whose own mode is `Delete` (running the unbinds, including the old `ex → orders-v1` unbind). This "create before destroy" ordering guarantees routing continuity regardless of the order the user writes the binding calls in the topology — in Update 1 the new bind is established before the old unbind runs, so the exchange is never without a binding mid-provisioning. Within each pass, a `RabbitMqQueueBindingDefinition` whose `QueueName` is in the `Delete`-queue set is skipped entirely (RabbitMQ auto-removes bindings on queue delete; an explicit unbind is redundant, and an explicit re-bind would target a queue that the preceding queue loop just deleted). A `Delete`-mode binding whose queue is **not** in the set is **not** skipped — its `Delete` arm runs the unbind, which is the Update-1 step that stops new messages flowing to the still-existing old queue. Provisioning order stays exchanges → queues → bindings, so by the time the binding loop runs, `Delete`-mode queues are already gone and the skip is correct.

### Compiler `Delete`-mode validation

In `ValidateConsumers`, after the existing "unknown queue" check, add: if the referenced queue's `DeclareMode == Delete`, emit "Inbound consumer for queue '…' references a queue declared with Delete mode; remove the Consume(…) call or change the queue's declare mode." The existing `Enum.IsDefined` checks in `ValidateExchangeDefinitions` / `ValidateQueueDefinitions` already accept the new `Delete` value without change (it is a defined enum member).

### Out of scope

Exchanges (evolutionary declaration, `Delete` provisioning, new knob methods, exchange-binding `Delete` provisioning), automatic physical-name migration / broker-side manifest / orphan auto-cleanup, `rabbitmqctl` / management HTTP API integration, and reading existing broker state to detect drift.
