## Rationale

BMF instruments the outbound path end to end: `OutboundDiagnostics` owns the `Bmf.Outbound` `ActivitySource`/`Meter`, `OutboundTarget` opens a `Producer` activity and records attempt/failure/duration metrics for every publish, and `TraceContextHeaders.Inject` writes the W3C `traceparent`/`tracestate`/baggage onto the wire (see `ai-plans/0001-8-publish-trace-and-span-ids.md`). The consumer side was explicitly deferred there.

The consequence is that the producer span is a dead end: the `traceparent` we publish is never read on receive, so every distributed trace breaks at the broker hop and the consumer has no span to link back to. There are also no inbound metrics, so consumer throughput, failures, and processing latency are invisible.

This plan closes that gap symmetrically to the outbound design: a transport-neutral core middleware that extracts the upstream trace context, opens a `Consumer` activity parented to it, and records inbound process metrics. It lands in `Bmf.Core` so every current and future transport is instrumented uniformly, exactly as `OutboundTarget` does for publishers. Adopting the OpenTelemetry `messaging.*` semantic conventions and shipping an OpenTelemetry registration package are addressed in a separate plan; this plan emits `bmf.inbound.*` attributes mirroring the existing `bmf.outbound.*` scheme so the two paths stay consistent.

## Acceptance Criteria

- [x] A new `InboundDiagnostics` static class (in `Bmf.Core/Messaging/Inbound/`) mirrors `OutboundDiagnostics`: it exposes an `ActivitySource` and `Meter` both named `Bmf.Inbound`, the process attempt/failure counters and the process duration histogram, and the public tag-name constants. Names follow the `bmf.inbound.*` convention.
- [x] A new framework middleware `InboundDiagnosticsMiddleware` (implementing `IMessageMiddleware`) is registered as the **outermost** stage of the inbound pipeline (before `FrameworkMessageAcknowledgementMiddleware`) by `RabbitMqTopologyCompiler.BuildPipeline`, and is registered in DI by `AddBmf`.
- [x] On each delivery the middleware extracts the upstream trace context from the transport headers (`context.Transport.Headers`) using `DistributedContextPropagator.Current` and starts a `Consumer` activity parented to it, so a consumer span links to the producer span across the broker. The extracted `tracestate` is applied to the activity (`Activity.TraceStateString`) and round-trips, and baggage carried on the wire is restored onto the activity.
- [x] The consumer activity is the current activity (`Activity.Current`) while the rest of the pipeline and the handler run, so any message the handler publishes is parented to the consumer span without further plumbing.
- [x] When no `traceparent` is present (e.g., a message published by a non-BMF producer, or with no active listener upstream) the middleware starts a new root `Consumer` activity and processing proceeds normally; when no listener is registered for `Bmf.Inbound`, `StartActivity` returns `null` and the middleware records metrics only, without allocating an activity.
- [x] The middleware records `bmf.inbound.process.attempts` once per delivery, `bmf.inbound.process.failures` on handler/pipeline failure, and `bmf.inbound.process.duration` (ms) always, tagged with the message type, endpoint name, source (queue), transport name, and outcome (`success` / `failure` / `cancelled`). The activity status is set to `Ok`/`Error` accordingly and the failure exception is recorded on the activity.
- [x] The `bmf.inbound.message.type` tag carries the CloudEvents **discriminator** (`context.Endpoint.Discriminator`), not the .NET type name, so it matches the value `bmf.outbound.message.type` carries on the producer side and inbound/outbound telemetry can be joined by message type. The .NET `Type` is kept off metrics to avoid a second, inconsistent name for the same thing.
- [x] A `cancelled` outcome (the delivery's `CancellationToken` was signaled, i.e., graceful shutdown) is distinguished from a `failure` outcome, mirroring the outbound `PublishDiagnostics` semantics, and does not increment the failure counter.
- [x] `TraceContextHeaders` is relocated to the direction-neutral `Bmf.Core.Messaging` namespace and gains public `Extract` support symmetric to `Inject`, so the automatic consumer path and raw-consume callers share one implementation; it honours `DistributedContextPropagator.Current` and is documented as the supported extraction helper.
- [x] Failures that occur in `RabbitMqTopologyRuntime.ProcessDeliveryAsync` *before* the pipeline runs (inspector / endpoint resolution — e.g. `UnknownInboundMessageException`) increment **both** `bmf.inbound.process.attempts` and `bmf.inbound.process.failures` from the runtime's general `catch (Exception)` block, so pre-pipeline rejects are counted without breaking the attempts ≥ failures invariant. The graceful-shutdown `catch (OperationCanceledException)` requeue path counts neither (it is a `cancelled`, not a `failure`, consistent with the middleware). (The full per-message span still begins at the middleware.)
- [x] XML doc comments are written for the new public surface; the three observability-contract types (`InboundDiagnostics`, `InboundDiagnosticsMiddleware`, `TraceContextHeaders.Extract`) get the verbose treatment specified under Technical Details → Documentation rather than one-line defaults, since they are the supported, externally consumed contract.
- [x] Automated tests are written.

## Technical Details

**`InboundDiagnostics` (new, `Bmf.Core/Messaging/Inbound/`).** A direct structural copy of `OutboundDiagnostics`:
- `ActivitySourceName = "Bmf.Inbound"`, `ActivitySource` and `Meter` over that name.
- Instruments: `bmf.inbound.process.attempts` (`Counter<long>`), `bmf.inbound.process.failures` (`Counter<long>`), `bmf.inbound.process.duration` (`Histogram<double>`, unit `ms`).
- Tag-name constants: `bmf.inbound.message.type`, `bmf.inbound.endpoint.name`, `bmf.inbound.source`, `bmf.inbound.transport.name`, `bmf.inbound.outcome`.

**`InboundDiagnosticsMiddleware` (new, `Bmf.Core/Messaging/Inbound/`).** Implements `IMessageMiddleware`. It mirrors the outbound outcome/duration *semantics*, but not the `PublishDiagnostics` struct: that struct exists to keep the publish hot path free of closure/delegate/state-machine allocations, and that rationale does not transfer to a middleware that is already an async state machine. A plain inline `try`/`catch`/`finally` over a `TagList` local is clearer and costs nothing extra here.
1. Build the base `TagList` from `context` (message type via `context.Endpoint.Discriminator` — the CloudEvents discriminator, matching the outbound `bmf.outbound.message.type`, **not** `context.MessageType.FullName`; endpoint name via `context.Endpoint.Name`; source/transport via `context.Transport.Source`/`context.Transport.TransportName`). Use `TagList` (stack-resident) rather than allocating an array per delivery.
2. Extract the parent context from `context.Transport.Headers` via the new `TraceContextHeaders.Extract`, then `InboundDiagnostics.ActivitySource.StartActivity("bmf.inbound.process", ActivityKind.Consumer, parentId)`. The `parentId` string overload restores `traceId`/`spanId` only, so set `activity.TraceStateString` from the extracted state explicitly (otherwise vendor trace state is dropped at every hop) and restore baggage. `using` the returned activity makes it `Activity.Current` for the awaited `next(context)`.
3. Record `process.attempts`, capture a `Stopwatch.GetTimestamp()`, `await next(context)`, then set outcome on the activity, and record `process.failures`/`process.duration` in an inline `try`/`catch (OperationCanceledException when token signalled)`/`catch`/`finally`, matching the outcome semantics of `OutboundTarget.PublishSerializedAsync`. The duration helper can mirror `OutboundTarget.GetDurationMilliseconds`.

Registered via `services.TryAddSingleton<InboundDiagnosticsMiddleware>()` in `AddBmf`, and added first in `RabbitMqTopologyCompiler.BuildPipeline` (`pipeline.UseMiddleware<InboundDiagnosticsMiddleware>()` before the acknowledgement middleware) so the span wraps acknowledgement, deserialization, user middleware, and the handler.

**`TraceContextHeaders` relocation + `Extract`.** Move `TraceContextHeaders` from `Bmf.Core.Messaging.Outbound` to the direction-neutral `Bmf.Core.Messaging` namespace and add `Extract` so a single class owns both directions. The move is a deliberate breaking change taken now (sanctioned for the pre-1.0 line by `AGENTS.md`) rather than after 0.1.0 ships: an `Extract` helper that is fundamentally about inbound consumption does not belong under `Outbound`, and the class is already transport- and direction-neutral. `Extract` calls `DistributedContextPropagator.Current.ExtractTraceIdAndState(...)` with a getter backed by `TransportMessage.TryGetHeaderString` (which already decodes the string / byte-array / memory header representations), returns the parent `traceparent`/`tracestate`, and exposes `ExtractBaggage` so the middleware can apply it. The signature accepts the inbound carrier shape (the `IReadOnlyDictionary<string, object?>` headers / `TransportMessage`), symmetric to the two `Inject` overloads.

**Pre-pipeline failures.** `RabbitMqTopologyRuntime.ProcessDeliveryAsync` already wraps inspection + endpoint resolution + `_topology.Pipeline(context)` in a `try` with two catches. In the **general `catch (Exception)`** block (the inspector/endpoint failure path, e.g. `UnknownInboundMessageException`), increment both `InboundDiagnostics.ProcessAttempts` and `InboundDiagnostics.ProcessFailures` (tagging what is known: source, transport). Incrementing attempts alongside failures here preserves the outbound invariant that every failure is also an attempt, so `failures/attempts` stays a clean ratio — the middleware, which owns the attempt count for everything that reaches the pipeline, is never reached on this path, so there is no double count. The **`catch (OperationCanceledException) when (stoppingToken.IsCancellationRequested)`** requeue path increments neither counter: a graceful-shutdown requeue is a `cancelled` outcome, not a failure, matching the middleware's outcome semantics. The per-message activity intentionally still starts inside the middleware (the inspector only reads headers); this keeps the span boundary transport-neutral while not losing the failure metric.

**Documentation.** Most members take ordinary one-line summaries, but three are the externally consumed observability contract and warrant more than the default — written to the bar `OutboundDiagnostics` already sets:
- **`InboundDiagnostics`** — the class summary names `Bmf.Inbound` as the `ActivitySource`/`Meter` to subscribe to in order to observe inbound telemetry (mirroring `OutboundDiagnostics`' "Subscribe to `ActivitySourceName`…" wording), and cross-references its outbound counterpart. Each public instrument and tag-name constant documents what it measures or annotates and its unit, so consumers can build dashboards from the XML docs alone.
- **`InboundDiagnosticsMiddleware`** — the type summary explains it is the consumer-hop span and inbound metrics recorder, that it parents the `Consumer` activity to the trace context extracted from the wire (this is what links it to the producer span across the broker), why it must be registered outermost so the span wraps acknowledgement and deserialization, and how it relates to the outbound `Producer` span. Note the `Activity.Current` side effect that re-parents handler-initiated publishes.
- **`TraceContextHeaders.Extract`** — mirror `Inject`'s existing `<remarks>`: these are W3C transport-propagation headers (OpenTelemetry messaging convention), distinct from and not to be confused with the `cloudEvents:*` attributes; it honours `DistributedContextPropagator.Current`; and it is the supported entry point for raw-consume callers that read trace context before the framework pipeline runs. Keep `Inject`'s CloudEvents-Distributed-Tracing-extension caveat consistent across the relocated class.

**Testing notes.** Register an `ActivityListener` (sampling `AllData`) on `Bmf.Inbound`, drive a delivery through the middleware with a known `traceparent` header, and assert the consumer activity's `ParentSpanId`/`TraceId` recover the producer context (re-derive via `DistributedContextPropagator.Current` rather than string-formatting). Add a `MeterListener` assertion for the three instruments across success / failure / cancellation. Extend the existing RabbitMQ Testcontainers integration test to assert an end-to-end trace: a published message's `TraceId` is recovered on the consumer span.
