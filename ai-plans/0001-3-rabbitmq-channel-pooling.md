## Rationale

`RabbitMqTarget<TMessage>.DispatchAsync` currently opens and disposes an `IChannel` per publish, which adds a network round trip to the hot path and caps publish throughput. Channel reuse is required, but `IChannel.BasicPublishAsync` is not safe for concurrent use, so reuse must be mediated by a pool that guarantees exclusive channel ownership per publish. The transport should offer two pooling topologies, defaulting to per-target ownership (which preserves per-target publish ordering at the default pool size of 1) and offering an opt-in connection-level shared pool for deployments that prioritize channel-count efficiency over per-target ordering. Channel-count math must be validated against the broker's advertised `channel_max` at first connection so misconfigurations fail fast instead of surfacing as runtime allocation failures.

## Acceptance Criteria

- [ ] `RabbitMqTarget<TMessage>` no longer creates and disposes an `IChannel` per publish. Publishes acquire a reusable channel from a transport-owned pool and release or discard it after the operation completes.
- [ ] The RabbitMQ transport supports two pooling modes selectable via `RabbitMqPublishingConfiguration`: `PerTarget` (default) and `Shared`.
- [ ] In `PerTarget` mode, each compiled target owns a bounded pool. The default `MaxChannelsPerTarget` is `1`, which preserves serialized publish behavior and per-target ordering. Configurable per transport, with the API shape leaving room for a per-route override in a future slice.
- [ ] In `Shared` mode, a single connection-level pool serves all targets. The default `SharedChannelPoolSize` is `8`. The configuration surface and XML doc comment on `Shared` mode must state that per-target publish ordering is not guaranteed in this mode.
- [ ] Pools grow lazily on demand up to their configured bound. When all channels are busy, callers wait asynchronously for one to be returned. Pools do not shrink during normal operation in this slice.
- [ ] A leased channel is never used concurrently by multiple publishers. The lease type guarantees exclusive ownership for the duration of the publish.
- [ ] If a pooled channel is closed, faulted, or otherwise unusable, it is discarded (not returned), the pool's live-channel count is decremented, and a later publish can create a replacement within the configured bound. A `BasicPublishAsync` exception that reflects a broker-side message rejection (channel still open) must not cause the channel to be discarded.
- [ ] At topology compile time, the transport computes and logs the worst-case channel count for the configured topology and mode.
- [ ] On first successful connection, the transport compares the worst-case channel count against `IConnection.ChannelMax` and throws a topology validation exception if the topology cannot fit. This check lives alongside the existing connection-acquisition path so it runs exactly once per connection lifecycle.
- [ ] Transport disposal tears down pooled channels before disposing the shared connection, without leaking channels or background work. Compiled targets and/or the shared pool participate in the transport's disposal flow explicitly.
- [ ] Automated tests are written.

## Technical Details

All pooling logic stays inside `Usf.Transport.RabbitMq`; `Usf.Core` remains unaware of channels and leases.

**Access modifiers.** No types or members introduced by this slice may use `internal`. Library consumers must have access to the same building blocks the transport itself uses. Lower-level APIs (the pool contract, the lease struct, the pool implementations) should follow the "public types hidden in plain sight" approach — public, but placed and named so that callers who only want the high-level publishing API are not tempted to reach for them. See https://blog.ploeh.dk/2015/09/21/public-types-hidden-in-plain-sight/.

**Lease abstraction.** Introduce a channel pool contract and a lease type (public, per the access-modifier rule above). The lease should be a `readonly struct` exposing the `IChannel`, allocation-free on the hot path. The return-vs-discard decision is centralized in the lease's dispose path — the caller never decides. On dispose the lease checks channel health (`IsOpen`, accounting for any observed `ChannelShutdown`) and returns a healthy channel to the pool or discards an unhealthy one. The pool's acquire path re-validates channel health on take and transparently discards-and-replaces any channel that closed asynchronously between return and next use. `RabbitMqTarget<TMessage>.DispatchAsync` becomes simply `await using var lease = await pool.AcquireAsync(ct); await lease.Channel.BasicPublishAsync(...)` — no error-handling branching in the target. A `BasicPublishAsync` exception that reflects a broker rejection (channel still open) correctly results in return, not discard, because the decision depends only on channel state.

**Pool primitive.** Back each pool with `System.Threading.Channels.Channel<IChannel>` (bounded). This gives async FIFO waiting and cancellation for free and is cleaner than `SemaphoreSlim + Queue<IChannel>`. The pool tracks live-channel count separately so it can create on demand up to the bound and decrement on discard.

**Pooling mode wiring.** Add a `RabbitMqChannelPoolingMode { PerTarget, Shared }` enum and the two integer options to `RabbitMqPublishingConfiguration`. In `PerTarget` mode, the topology compiler constructs one pool per compiled target and passes it into the target's constructor; targets hold their pool directly. In `Shared` mode, the compiler constructs a single connection-level pool registered with the transport and passes the same pool reference into every compiled target. The target implementation is agnostic to which mode produced its pool — it only sees an `IRabbitMqChannelPool`-shaped dependency. Storing the pool reference per target (rather than resolving it at publish time) keeps a future per-route override one constructor argument away.

**Channel creation and connection.** The pool obtains channels via the existing `RabbitMqConnectionManager.GetConnectionAsync`. If channel creation fails because the underlying connection is dead, the pool must propagate the failure cleanly so a future reconnect slice can plug in. No broader reconnect-policy changes belong in this slice.

**Channel-count validation.** Add a method on `RabbitMqConnectionManager` (or a small helper invoked from it) that runs once on first connection and validates worst-case channel count against `IConnection.ChannelMax`. Worst-case is `targets × MaxChannelsPerTarget` in `PerTarget` mode and `SharedChannelPoolSize` in `Shared` mode. The startup log line should be specific enough to be actionable — e.g., `"RabbitMQ publish topology may open up to 14 channels (PerTarget mode, 14 targets × max 1)."`

**Disposal.** Targets become `IAsyncDisposable` in `PerTarget` mode (they own their pool); the compiled topology disposes targets during its own disposal. `RabbitMqCompiledTopology` itself must implement both `IAsyncDisposable` and `IDisposable` so DI-driven shutdown can drive the tear-down. In `Shared` mode the transport disposes the shared pool. Either way, pool disposal must complete before `RabbitMqConnectionManager` tears down the connection. The hosted-service shutdown flow needs to be checked end-to-end to ensure the disposal order holds.

**Tests.** In addition to the standard behavioral checks (sequential reuse, concurrent isolation, bounded growth + waiting, faulted-channel replacement, clean disposal), include: a test that channel-count validation throws when the topology exceeds a simulated `ChannelMax`, and a test that a fault mid-publish does not leak the lease slot (the live-count decrement happens on the exception path).
