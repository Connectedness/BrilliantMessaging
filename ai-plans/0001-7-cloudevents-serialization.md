## Rationale

`SerializedMessage` carries a body, content-type, headers, message-id, and correlation-id, but nothing on the wire says *what contract* a message is. `Utf8JsonMessageSerializer` even hardcodes `MessageId`/`CorrelationId` to `null`, so today a published message is an anonymous blob of JSON. The moment inbound consumers land they must know how to deserialize what the publisher wrote, and if outbound ships without a stable envelope convention we buy a breaking wire-format change the day inbound arrives. This is the single most important forward-compatibility item even though it is invisible while only outbound exists.

This slice settles that convention by making USF opinionated about CloudEvents. Every published message becomes a CloudEvent in **binary content mode**: the data section stays the raw JSON payload we serialize today, and the CloudEvents attributes ride alongside as transport metadata following the [CloudEvents AMQP Protocol Binding](https://github.com/cloudevents/spec/blob/v1.0.2/cloudevents/bindings/amqp-protocol-binding.md) (reused over AMQP 0.9.1). The contract identity — the CloudEvents `type` — is an explicitly declared, stable string decoupled from CLR type names, held in a registry that is *asymmetric* so it can serve both serialization now and backwards-compatible deserialization later. We do **not** depend on `Light.PortableResults`; its structured-CloudEvents support can later plug in as an alternative payload/serializer behind the same seams.

Two deliberate constraints shape the design. First, the `type` must never be a CLR `FullName` (a rename or namespace move would silently break every consumer), so it is declared and stored in a registry. Second, retry-sensitive attributes (`id`, `time`) must come from the call site and never be (re)generated during serialize/publish — a publish retried after an ambiguous NACK or confirm-timeout must be the same logical event, or broker/consumer dedup sees a duplicate. Generating `id`/`time` at *message construction* (the call site) is fine and stable; generating them inside the serializer is not. Structured content mode, distributed trace-context propagation, and inbound deserialization are explicitly out of scope here; this slice only fixes the outbound wire envelope so those can follow without a breaking change.

## Acceptance Criteria

- [x] A new `Usf.Abstractions` project (netstandard2.0) is created so contract assemblies can reference the message-shape types without dragging in `Usf.Core`. It is added to `USF.slnx` under `/src/` and referenced by `Usf.Core`. It contains: `ICloudEvent`, the `BaseCloudEvent` abstract base type, a `UsfUuid.NewId()` id helper, and the transport-neutral CloudEvents attribute-name constants. Its only runtime third-party dependency is the small, dependency-free [`Ulid`](https://github.com/Cysharp/Ulid) package; language/runtime polyfills (`init`, records, nullable attributes) come from [`Polyfill`](https://github.com/SimonCropp/Polyfill), a source-only (`PrivateAssets=all`) package that adds no runtime or transitive dependency.
- [x] netstandard2.0 projects use [`Polyfill`](https://github.com/SimonCropp/Polyfill) (source-only, `PrivateAssets=all`) for language/runtime polyfills in place of `IsExternalInit`. The new `Usf.Abstractions` references `Polyfill`, and the `IsExternalInit` reference is removed from `Directory.Packages.props` and from the projects this slice already touches (`Usf.Core`, `Usf.Transport.RabbitMq`), which switch to `Polyfill`. Because it is source-only it adds no runtime or transitive dependency for consumers.
- [x] `ICloudEvent` exposes the call-site-owned, retry-stable per-instance attributes: a non-empty `Id` (`Guid`) and `Time` (`DateTimeOffset`), plus an optional `Subject`. It does **not** expose `Type` (that is contract-level, resolved from the registry) or `Source` (that is application-level). The id is a `Guid` on the .NET surface; the CloudEvents `id` attribute is a string on the wire (see `CloudEventEnvelope`).
- [x] `BaseCloudEvent` implements `ICloudEvent` and defaults `Id` to a new **time-ordered `Guid`** (via `UsfUuid.NewId()`) and `Time` to `DateTimeOffset.UtcNow` **at construction**. A time-ordered id gives index locality in outbox/dedup stores and natural sortability — preferred over a random v4 `Guid` for a messaging library. The XML docs state explicitly that this construction-time generation is the call site supplying the value (stable for the instance and across retries), and is distinct from — and must not be moved into — serialize/publish-time generation, which is forbidden.
- [x] `UsfUuid.NewId()` returns a time-ordered `Guid` generated via `Ulid.NewUlid().ToGuid()` (Cysharp/Ulid), which works natively on `netstandard2.0` so the package stays single-target. Note this is a ULID projected to a `Guid` — time-ordered like UUIDv7 but not bit-for-bit RFC-9562 v7 (the version/variant nibbles differ); this is immaterial because the CloudEvents `id` is an opaque string. Centralizing generation behind `UsfUuid.NewId()` keeps the choice swappable (e.g. to native `Guid.CreateVersion7()` if the package later targets `net9.0`+).
- [x] The CloudEvents `type` is an explicitly declared stable string, never derived from a CLR type name. It is declared only via explicit registration against the contract registry builder (no attribute, no reflection scanning).
- [x] An `IMessageContractRegistry` provides an **asymmetric** mapping: serialize resolves exactly one canonical discriminator per CLR type (`Type → string`); deserialize resolves many discriminators to one CLR type (`string → Type`), so an old and a new discriminator can both map to the same type for backwards-compatible deserialization. Registration is one canonical discriminator (used both ways) plus zero-or-more inbound-only aliases.
- [x] The registry is validated and built **fail-fast at startup**, enforcing globally: no discriminator string maps to two different CLR types (the primary safety net), and every registered type has at most one canonical discriminator. The round-trip invariant (`deserialize(serialize(x))` resolves back to the same type) is enforced **conditionally** — only for a type that has inbound discriminators registered, in which case its canonical serialize discriminator must be among them. A `Map<T>("x")` call seeds both directions, so the common publish-and-consume case satisfies this by construction; a **publish-only** service (a type with no inbound registration, or a service that registers no consume side at all) is exempt and must not fail startup. Violations throw a dedicated registration/validation exception listing all offending entries.
- [x] A startup cross-check validates the registry against the outbound topology: every outbound target that declares a `MessageType` must have a registered canonical discriminator, otherwise startup fails fast with a clear error naming the unregistered message type(s) and target name(s). A forgotten registration surfaces at startup, not on first publish. (Raw-only targets, whose `MessageType` is `null`, are exempt.)
- [x] `IMessageSerializer` is reshaped to target CloudEvents directly: `SerializeAsync<T>(T message, in CloudEventMetadata metadata, CancellationToken)` returns a structured `CloudEventEnvelope` (the resolved attributes plus the serialized data bytes), not an anonymous `SerializedMessage`. The serializer owns CloudEvents *attribute assembly*; the *protocol binding* (how attributes map to a concrete transport message) is the transport's responsibility.
- [x] The existing `Utf8JsonMessageSerializer` is **removed**, not adapted: it produces envelope-less, anonymous messages (null `MessageId`/`CorrelationId`, empty headers) that cannot satisfy the new CloudEvents contract, and leaving a non-CloudEvents `IMessageSerializer` on the normal path would reopen the bypass the envelope convention exists to close. Its declared-vs-runtime `JsonTypeInfo` polymorphism logic is preserved by migrating it into `Utf8JsonPayloadCodec`. Existing tests targeting `Utf8JsonMessageSerializer` are migrated to cover `Utf8JsonPayloadCodec` and `CloudEventMessageSerializer`.
- [x] Payload encoding (data section → bytes + `datacontenttype`) is factored behind a narrow `IPayloadCodec` collaborator so the CloudEvents envelope-assembly logic is written once across present and future codecs. `Utf8JsonPayloadCodec` carries over the existing declared-vs-runtime `JsonTypeInfo` polymorphism handling (`ShouldUseWith`/`HasKnownPolymorphism`). This is a focused strategy, not an `IMessageSerializer`-over-`IMessageSerializer` decorator chain.
- [x] The single, opinionated `CloudEventMessageSerializer : IMessageSerializer` resolves attributes by layer: `type` from the registry (by the message's runtime type, consistent with the codec's polymorphism choice); `id`/`time`/`subject` from `CloudEventMetadata`; `source` from configured `CloudEventsOptions` (with optional per-call override); `specversion` constant `1.0`; `datacontenttype` from the codec. It never generates `id` or `time`, and treats a `Guid.Empty` id as missing. The id is rendered to the wire `id` string in canonical lowercase form (`Guid.ToString("D")`). If the required set (`id`, `time`, `source`, `type`) cannot be resolved it throws a dedicated `CloudEventMetadataException` whose message names the missing attribute and how to supply it (implement `ICloudEvent`/derive from `BaseCloudEvent`, pass `CloudEventMetadata`, register the type's discriminator, or configure `Source`).
- [x] `IMessagePublisher` exposes two typed publish overloads: one constrained `where T : ICloudEvent` that reads `id`/`time`/`subject` from the message (the idiomatic pit-of-success path), and one unconstrained overload taking `in CloudEventMetadata` for message types that cannot implement the interface (third-party, source-generated). Both flow the metadata into `OutboundTarget<T>`/the serializer. `PublishRawAsync(SerializedMessage, ...)` remains the unchanged low-level escape hatch (the caller owns the full wire representation, CloudEvents or not).
- [x] The RabbitMQ transport applies the CloudEvents AMQP binary binding when publishing a `CloudEventEnvelope`: `datacontenttype` → AMQP `content-type`; every remaining attribute (`id`, `specversion`, `source`, `type`, `time`, `subject`, any extensions) → application headers prefixed `cloudEvents:` (the prefix lives in the transport, since the binding is transport-specific); and `id` is additionally copied to AMQP 0.9.1 `MessageId` as a convenience mapping. The data bytes stay the message body. Per-target route headers/routing keys continue to apply on top. The raw `SerializedMessage` path keeps its current `BasicProperties` mapping.
- [x] The publish diagnostics message-type tag (currently `Type.FullName` via `MessagePublisher.GetMessageTypeName`) is the resolved CloudEvents discriminator, so telemetry is keyed by the wire contract rather than the CLR type name. The raw `PublishRawAsync` path, which has no resolved discriminator, keeps its existing `serialized`/`MessageType`-derived tag value.
- [x] `CloudEventsOptions` holds the application `Source` (a non-empty URI-reference) and is validated at startup; `CloudEventMessageSerializer`, `IPayloadCodec`, and `IMessageContractRegistry` are wired into DI where `IMessageSerializer` is resolved today.
- [x] XML doc comments document the wire convention as "CloudEvents v1.0, binary content mode, AMQP protocol binding over AMQP 0.9.1", the retry-stability rule for `id`/`time`, and the asymmetric registry's backwards-compatible-deserialization purpose.
- [x] Automated tests are written. Integration tests assert the on-the-wire `BasicProperties` (content-type, `MessageId`, `cloudEvents:*` headers, body unchanged) against RabbitMQ via Testcontainers.

## Technical Details

**`Usf.Abstractions` (new, netstandard2.0).** The one real cost of "every message implements our interface" is coupling a shared contract assembly to the heavy `Usf.Core`; a featherweight abstractions package (the MassTransit.Abstractions / MediatR.Contracts pattern) removes it. It depends only on the source-only `Polyfill` package (for `init`/record/nullable-attribute support on netstandard2.0, `PrivateAssets=all`, no runtime/transitive dependency) and the small, dependency-free `Ulid` package. `ICloudEvent` carries `Guid Id { get; }`, `DateTimeOffset Time { get; }`, `string? Subject => null`. `BaseCloudEvent` is the abstract base implementing it with `Id` (a `UsfUuid.NewId()` time-ordered `Guid`) and `Time` (`UtcNow`) defaulted at construction. `UsfUuid.NewId()` wraps `Ulid.NewUlid().ToGuid()`: ULID generation runs natively on netstandard2.0, so the package stays single-target and consumers on .NET Framework still get a time-ordered id rather than a random v4. The wrapper is the single swap-point if the project later wants native `Guid.CreateVersion7()`. The attribute-name constants (`id`, `source`, `type`, `specversion`, `time`, `subject`, `datacontenttype`, `dataschema`) live here as the shared, transport-neutral vocabulary that the future inbound side also keys off; the AMQP `cloudEvents:` header prefix deliberately does **not** live here — it is an AMQP-binding concern owned by the RabbitMQ transport (a future Kafka transport would use its own `ce_` prefix).

**Contract registry (`Usf.Core`).** `IMessageContractRegistry` exposes `string GetDiscriminator(Type)` (serialize, throws if unmapped) and `bool TryResolveType(string, out Type)` (deserialize). Back it with a `Dictionary<Type,string>` (one canonical entry per type) and a `Dictionary<string,Type>` (many keys → one type). A builder collects registrations via explicit `Map<T>("canonical").WithInboundAlias("legacy")` calls (optionally a `dataschema` per type) — and runs the fail-fast validation (global collision and single-canonical checks, plus the conditional round-trip check for types that have inbound discriminators) when built, aggregating all errors into one dedicated exception in the spirit of `OutboundTopologyValidationException`. Publish-only registration is first-class: a service that only ever calls `Map<T>` for the types it publishes builds a valid registry with no consume side. The registry and its builder live in `Usf.Core` (not Abstractions): contract assemblies need only `ICloudEvent`/`BaseCloudEvent`, not the machinery, and the discriminators are declared at composition time alongside the rest of the messaging configuration. Discriminator resolution uses the message's runtime type so it agrees with the codec's declared-vs-runtime polymorphism decision.

**Startup cross-check.** Independent of the registry's own build-time validation, a startup step reconciles the registry with the outbound topology: it enumerates the topology's typed targets (every `OutboundTarget` whose `MessageType` is non-null) and asserts each `MessageType` resolves to a canonical discriminator via `IMessageContractRegistry`. `OutboundTopology` currently exposes only keyed lookups, so this needs a way to enumerate the registered typed message types — either a small read-only enumeration on `IOutboundTopology`/`OutboundTopology` or running the check at composition time where the full target set is already in hand. Wire it into the existing startup path (e.g. `OutboundTopologyHostedService.StartAsync`, before provisioning, or an equivalent validated-options step) so unregistered outbound contracts fail fast at startup with all offenders aggregated into one error rather than throwing from `GetDiscriminator` on first publish. Raw-only targets (`MessageType == null`) are skipped.

**Serialization reshape (`Usf.Core.Messaging.Serialization`).** Today's `Utf8JsonMessageSerializer` is deleted and its responsibilities are redistributed across two new types. `IPayloadCodec` is the narrow data-section strategy — `EncodedPayload Encode<T>(T message)` returning bytes plus `datacontenttype` (e.g. `application/json`) — and `Utf8JsonPayloadCodec` carries over the existing `ShouldUseWith`/`HasKnownPolymorphism`/`JsonTypeInfo` logic verbatim. `CloudEventMessageSerializer : IMessageSerializer` is the only normal-path serializer; it calls the codec for the data section, resolves the CloudEvents attributes by the layered strategy in the acceptance criteria, and returns a `CloudEventEnvelope`. `CloudEventEnvelope` is a `readonly record struct` holding `SpecVersion`, `Id` (the canonical lowercase `Guid.ToString("D")` rendering — a `string`, because the CloudEvents `id` attribute is a string and inbound interop must accept non-GUID ids), `Source`, `Type`, `Time`, `Subject?`, `DataContentType`, `DataSchema?`, the `Data` bytes, and an optional extensions dictionary. `CloudEventMetadata` is a `readonly record struct` carrying the call-site `Id` (`Guid`), `Time`, optional `Subject`, and optional `Source` override; the serializer rejects `Guid.Empty` as a missing id. `IMessageSerializer.SerializeAsync<T>` changes signature to take `in CloudEventMetadata` and return `ValueTask<CloudEventEnvelope>`; this is a breaking change, acceptable pre-release.

**Why the binding is transport-side.** The CloudEvents protocol binding (which attribute becomes which transport field, and the header prefix) is genuinely transport-specific, so the serializer stops at structured attributes and each transport binds them. This keeps the `cloudEvents:` AMQP prefix out of `Usf.Core` and lets a future transport bind differently without touching core. Consequently `OutboundTarget<T>` threads the envelope, not a flattened `SerializedMessage`, to the transport.

**`OutboundTarget` changes.** `OutboundTarget<T>.PublishAsync` gains the `CloudEventMetadata` (supplied by the publisher), serializes to a `CloudEventEnvelope`, and calls a new typed hook — rename `PublishTypedSerializedAsync(message, serializedMessage, ct)` to `PublishTypedCloudEventAsync(message, envelope, ct)` (the `message` is still needed for message-derived routing keys/headers). The base `OutboundTarget.PublishSerializedAsync(SerializedMessage, ct)` stays as-is for the raw escape hatch. The existing serialize-failure wrapping into `MessageSerializationException` is unchanged.

**RabbitMQ transport.** `RabbitMqOutboundTarget<TMessage>` implements `PublishTypedCloudEventAsync` by building `BasicProperties` from the envelope per the binary binding (a new `CreateBasicProperties(CloudEventEnvelope, routeHeaders)` overload): `ContentType = envelope.DataContentType`, `MessageId = envelope.Id` as a convenience mapping for AMQP 0.9.1, and a header per remaining CloudEvents attribute — including `id` — under the `cloudEvents:` prefix (string values, encoded as RabbitMQ longstr), merged with the per-target route headers exactly as `CreateBasicProperties` does today; body is `envelope.Data`. `PublishSerializedAsync(SerializedMessage)` keeps its current mapping for raw publishes. `DispatchAsync`, channel leasing, publisher-confirm, and reconnect paths are untouched — and the retry/confirm machinery must continue to reuse the same envelope on retry so `id`/`time` stay stable (no regeneration), which holds for free because the envelope is built once before dispatch.

**Publisher.** `IMessagePublisher` gains `PublishMessageAsync<T>(T, OutboundTarget?, CancellationToken) where T : ICloudEvent` (extracts metadata from the message) and `PublishMessageAsync<T>(T, in CloudEventMetadata, OutboundTarget?, CancellationToken)` (explicit metadata). Both resolve the target and call `OutboundTarget<T>.PublishAsync` with the metadata. The constraint is enforced at compile time only on the convenience overload; the unconstrained overload keeps the door open for types that cannot implement `ICloudEvent` without a future breaking change. `MessagePublisher` replaces `Type.FullName` (today's `GetMessageTypeName`) with the resolved CloudEvents discriminator for the message-type tag so telemetry is keyed by the wire contract; the discriminator is already resolved for the envelope, so it is threaded into `PublishWithDiagnosticsAsync` rather than recomputed. The raw path keeps its existing `serialized`/`MessageType` tag value (no discriminator is resolved there). The rest of the diagnostics flow is unchanged.

**DI / options.** `CloudEventsOptions` (required non-empty `Source`, validated at startup via options validation) is registered alongside the messaging services; `CloudEventMessageSerializer`, `Utf8JsonPayloadCodec`, and the built `IMessageContractRegistry` replace the current `IMessageSerializer` registration. Source resolution is configuration-first with an optional per-call override on `CloudEventMetadata`; an ambient `source`/context accessor is intentionally deferred (trace-context propagation is out of scope here).

Out of scope, by design: structured content mode (`application/cloudevents+json`) — deferred and the seam where `Light.PortableResults` later plugs in; inbound deserialization — but the asymmetric registry and attribute-name constants are built now precisely so inbound reuses them unchanged; distributed trace-context (`traceparent`) propagation; and any dependency on `Light.PortableResults`.
